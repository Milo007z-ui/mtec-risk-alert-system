## 1. ทฤษฎีที่เกี่ยวข้อง

ระบบนี้อาศัยองค์ความรู้หลายด้านร่วมกัน ทั้งด้านวิศวกรรมความปลอดภัยจราจร
การวิเคราะห์ข้อมูลเชิงพื้นที่ และเทคโนโลยีเว็บ/ระบบฝังตัว โดยมีทฤษฎีหลักดังนี้

### 1.1 การหาระยะทางบนพื้นผิวโลก (Haversine Formula)

หัวใจของการแจ้งเตือนคือการรู้ว่า "ผู้ใช้อยู่ห่างจากจุดเสี่ยงกี่เมตร" เนื่องจาก
พิกัด GPS อยู่บนทรงกลม (ละติจูด/ลองจิจูด) จึงใช้สูตร **Haversine** คำนวณระยะทาง
วงกลมใหญ่ (great-circle distance) ระหว่างสองพิกัด:

$$d = 2R \cdot \arcsin\left(\sqrt{\sin^2\left(\tfrac{\Delta\varphi}{2}\right) + \cos\varphi_1 \cos\varphi_2 \sin^2\left(\tfrac{\Delta\lambda}{2}\right)}\right)$$

โดย $R$ = รัศมีโลก (≈ 6,371,000 ม.), $\varphi$ = ละติจูด, $\lambda$ = ลองจิจูด
ระบบใช้สูตรนี้ทั้งบนเว็บ (`js/distance.js`) และบนเซิร์ฟเวอร์ API เพื่อคัดกรองจุดเสี่ยง
ที่อยู่ในรัศมีที่กำหนด ก่อนคำนวณระยะจริงเพื่อประหยัดการประมวลผล (bounding-box pre-filter)

### 1.2 การจัดกลุ่มเชิงพื้นที่ด้วย DBSCAN

ข้อมูลอุบัติเหตุดิบเป็น "จุดเกิดเหตุรายครั้ง" หลายพันจุด ซึ่งกระจัดกระจาย
จำเป็นต้องรวมเหตุการณ์ที่เกิดในบริเวณเดียวกันให้เป็น "จุดเสี่ยง" เดียว โดยใช้อัลกอริทึม
**DBSCAN** (Density-Based Spatial Clustering of Applications with Noise) ซึ่งมีข้อดีคือ

- ไม่ต้องกำหนดจำนวนกลุ่มล่วงหน้า (ต่างจาก K-Means)
- จัดกลุ่มตามความหนาแน่น จุดโดดเดี่ยว (noise) ถูกคัดออกอัตโนมัติ
- ใช้ระยะทาง **haversine** เป็น metric ได้โดยตรง

พารามิเตอร์ที่ใช้: `eps = 400 เมตร` (รัศมีความเป็นเพื่อนบ้าน) และ
`min_samples = 3` (ต้องมีอย่างน้อย 3 อุบัติเหตุจึงนับเป็นจุดเสี่ยง) — ค่านี้ปรับให้เหมาะกับ
ถนนในเขตเมืองที่หนาแน่น เพื่อไม่ให้เกิดจุดเสี่ยงถี่เกินไปจนแจ้งเตือนบ่อยเกินเหตุ

### 1.3 การประเมินความรุนแรง EPDO และหลักการ KSI

การนับ "จำนวนครั้ง" อย่างเดียวไม่สะท้อนอันตรายจริง จุดที่เกิดเหตุน้อยครั้ง
แต่มีผู้เสียชีวิต ย่อมอันตรายกว่าจุดที่ชนกันบ่อยแต่แค่เฉี่ยว จึงใช้ดัชนี **EPDO**
(Equivalent Property Damage Only) ตามแนวปฏิบัติของ FHWA/HSIP แปลงผู้ประสบเหตุ
แต่ละระดับเป็นหน่วยความสูญเสียเทียบเท่า:

$$\text{EPDO} = (\text{เสียชีวิต} \times 10) + (\text{บาดเจ็บสาหัส} \times 4) + (\text{บาดเจ็บเล็กน้อย} \times 1)$$

และใช้หลักการ **KSI** (Killed or Seriously Injured) เป็น safety override —
จุดใดที่เคยมีผู้เสียชีวิตจะถูกยกระดับความเสี่ยงขึ้นทันที ไม่ให้ถูกจัดเป็นระดับต่ำ
แม้คะแนนรวมจะไม่สูงก็ตาม

### 1.4 แบบจำลองประเมินความเสี่ยงถนน (Road Risk Assessment Model — เต็ม 100)

แต่ละจุดเสี่ยงได้คะแนน `risk_score` จากการถ่วงน้ำหนัก 4 ปัจจัย:

| ปัจจัย | น้ำหนัก | หลักการ |
|---|---|---|
| Accident Frequency (ความถี่) | 30 | จำนวนอุบัติเหตุ ปรับสเกลแบบ log เพราะข้อมูลมีการแจกแจงหางยาว (long-tail) อิ่มตัวที่ 50 ครั้ง/ปี |
| Accident Severity (ความรุนแรง) | 35 | ค่า EPDO ปรับสเกล log อิ่มตัวที่ 120 (≈ เสียชีวิต 12 ราย) |
| Road Geometry (ลักษณะถนน) | 20 | ค่าเฉลี่ยถ่วงน้ำหนักอันตรายตามจำนวน "จุดขัดแย้งกระแสจราจร" (conflict points): ทางแยก 1.0 > จุดกลับรถ 0.9 > ทางโค้ง 0.8–0.9 > ทางตรง 0.3 |
| Speed Limit (ความเร็ว) | 15 | $(\text{ความเร็วจำกัด}/120)^2$ เพราะพลังงานจลน์แปรผันตามความเร็วยกกำลังสอง |

**การใช้สเกลลอการิทึม (log scaling):** เนื่องจากจำนวนอุบัติเหตุกระจายแบบหางยาว
(จุดส่วนใหญ่ 3–20 ครั้ง แต่จุดสูงสุดถึง 364 ครั้ง) การใช้สเกลเชิงเส้นจะทำให้จุดสุดโต่ง
กลบจุดอื่นทั้งหมด จึงใช้ log เพื่อให้ความแตกต่างช่วงต้นมีน้ำหนักและอิ่มตัวที่ค่าสูง

### 1.5 การแบ่งระดับความเสี่ยงด้วยควอร์ไทล์ (Quartile Calibration)

แทนที่จะตั้งเกณฑ์ตายตัว 70/40 ระบบคาลิเบรตเกณฑ์จากการกระจายคะแนนจริง
(min 27, มัธยฐาน ≈ 45.8, max 83.5) โดยตัดที่ควอร์ไทล์บน เพื่อควบคุมสัดส่วนจุดสีแดง
ไว้ราว 25% ลดปัญหา **alert fatigue** (การเตือนบ่อยจนผู้ขับเพิกเฉย):

- 🟥 **สูง** — score ≥ 55 *หรือ* เสียชีวิต ≥ 2 ราย
- 🟧 **ปานกลาง** — score ≥ 40 *หรือ* เสียชีวิต ≥ 1 ราย
- 🟩 **ต่ำ** — ที่เหลือ

### 1.6 ระบบระบุตำแหน่ง (GPS) และ Geolocation API

- **บนเว็บ/มือถือ:** ใช้ HTML5 Geolocation API (`navigator.geolocation.watchPosition`)
  ซึ่งบังคับให้ทำงานเฉพาะบน HTTPS หรือ localhost เพื่อความปลอดภัย
- **บนอุปกรณ์ฝังตัว:** อ่านพิกัดจากโมดูล GPS ผ่าน **gpsd** (GPS daemon) ด้วยโปรโตคอล
  TCP/JSON โดยสนใจเฉพาะรายงานชนิด TPV (Time-Position-Velocity)

### 1.7 การสังเคราะห์เสียงพูดภาษาไทย (Text-to-Speech)

เพื่อให้ผู้ขับไม่ต้องละสายตาจากถนน ระบบแจ้งเตือนด้วยเสียงพูดแทนการอ่านหน้าจอ:

- **บนเว็บ:** Web Speech API (`speechSynthesis`) ด้วยเสียงภาษาไทย `th-TH`
- **บนอุปกรณ์:** `espeak-ng` ซึ่งรองรับภาษาไทย (ปรับความเร็วพูดช้าลงเป็น 140 wpm
  ให้ฟังชัดขณะขับรถ)

### 1.8 ตรรกะป้องกันการเตือนซ้ำ (Hysteresis & Cooldown)

หากเตือนทุกครั้งที่อยู่ในรัศมี ระบบจะเตือนรัวขณะรถติดอยู่ใกล้จุดเสี่ยง จึงใช้หลักการ
**hysteresis** (มีสองขอบเขต) ร่วมกับ cooldown:

- เตือนครั้งแรกเมื่อเข้าสู่รัศมี **500 ม.** (`ALERT_RADIUS_M`)
- จะเตือนจุดเดิมซ้ำได้ก็ต่อเมื่อออกไกลกว่า **600 ม.** (`EXIT_RADIUS_M`) แล้วกลับเข้ามาใหม่
  หรือวนอยู่ในรัศมีนานเกิน **5 นาที** (`REALERT_MS`)

ขอบเขตขาเข้า (500) กับขาออก (600) ที่ต่างกันช่วยกันการ "เด้ง" เตือนซ้ำตรงขอบรัศมี

### 1.9 สถาปัตยกรรม REST API แบบ Client–Server

ระบบออกแบบให้แยกส่วนข้อมูล (จุดเสี่ยง) ออกจากส่วนแสดงผล ผ่าน **REST API**
(สถาปัตยกรรม client–server, HTTP GET, ข้อมูล JSON) ทำให้อุปกรณ์ปลายทางหลายชนิด
(เว็บ, มือถือ, Raspberry Pi) ดึงข้อมูลชุดเดียวกันไปใช้ได้ และปรับปรุงกติกาแจ้งเตือน
ที่ศูนย์กลางเพียงจุดเดียว

## 2. การเลือกอุปกรณ์

การเลือกอุปกรณ์ยึดหลัก **ต้นทุนต่ำ ติดตั้งบนรถได้จริง และใช้ซอฟต์แวร์โอเพนซอร์ส**
เพื่อให้ต่อยอดใช้กับรถสาธารณะ (เช่น รถเมล์) ได้ในวงกว้าง

### 2.1 หน่วยประมวลผลปลายทาง — Raspberry Pi

| ประเด็น | เหตุผลที่เลือก |
|---|---|
| ราคา/ขนาด | ราคาถูก ขนาดเล็ก กินไฟต่ำ ติดตั้งในรถได้สะดวก |
| ระบบปฏิบัติการ | รัน Linux (Raspberry Pi OS) รองรับ `gpsd` และ `espeak-ng` ได้ทันที |
| การเชื่อมต่อ | มีพอร์ต USB สำหรับ GPS, ช่องเสียง 3.5 มม./HDMI/USB สำหรับลำโพง, Wi-Fi/4G สำหรับเรียก API |
| ซอฟต์แวร์ | ไคลเอนต์เขียนด้วย Python มาตรฐาน (standard library) ล้วน ไม่ต้องติดตั้งไลบรารีเพิ่ม ลดภาระการดูแลในภาคสนาม |

### 2.2 โมดูลรับสัญญาณ GPS (USB GPS Receiver)

- เลือกแบบ **USB** ที่ทำงานผ่าน `gpsd` ได้ เพราะเป็นมาตรฐานเปิด รองรับอุปกรณ์หลายรุ่น
- ระบบอ่านพิกัดผ่านโปรโตคอล gpsd (TCP/JSON) จึงไม่ผูกกับฮาร์ดแวร์ยี่ห้อใดยี่ห้อหนึ่ง
- รองรับการทำงานแม้สัญญาณดาวเทียมขาดหาย โดยใช้ค่าพิกัดล่าสุด (last fix) ต่อไปจนกว่า
  จะได้สัญญาณใหม่

### 2.3 ลำโพง/ชุดขยายเสียง

- ต่อผ่านช่องเสียง 3.5 มม. หรือ USB ของ Raspberry Pi
- ใช้แจ้งเตือนด้วยเสียงพูด (`espeak-ng`) ตั้งความเร็วพูดช้าลงเพื่อความชัดเจนขณะขับรถ

### 2.4 เครื่องแม่ข่าย (Server) สำหรับ REST API

- ใช้ **FastAPI** (Python) เพราะเบา ประสิทธิภาพสูง และสร้างเอกสาร API อัตโนมัติ
  (Swagger UI ที่ `/docs`) ช่วยให้ทดสอบ/ต่อยอดง่าย
- เซิร์ฟเวอร์เดียวทำหน้าที่ทั้งแจกจ่ายข้อมูลจุดเสี่ยงและเสิร์ฟหน้าเว็บไปพร้อมกัน

### 2.5 อุปกรณ์ทางเลือก — สมาร์ตโฟน/เบราว์เซอร์

- สำหรับผู้ใช้ทั่วไปที่ไม่มีอุปกรณ์ฝังตัว สามารถใช้ผ่านเว็บเบราว์เซอร์บนมือถือได้ทันที
  โดยอาศัย GPS และลำโพงในตัวเครื่อง ไม่ต้องลงแอป

### สรุปรายการอุปกรณ์

| อุปกรณ์ | หน้าที่ | ซอฟต์แวร์ที่เกี่ยวข้อง |
|---|---|---|
| Raspberry Pi | หน่วยประมวลผลบนรถ | `device/pi_alert_client.py` |
| USB GPS Receiver | รับพิกัดตำแหน่งรถ | `gpsd` |
| ลำโพง | แจ้งเตือนด้วยเสียงพูด | `espeak-ng` |
| เครื่องแม่ข่าย | แจกจ่ายข้อมูลจุดเสี่ยง | FastAPI (`api/server.py`) |
| สมาร์ตโฟน (ทางเลือก) | ใช้งานผ่านเบราว์เซอร์ | Geolocation + Web Speech API |

## 3. การดำเนินการ

การพัฒนาระบบดำเนินตามขั้นตอนต่อไปนี้

### ขั้นที่ 1 — รวบรวมข้อมูลอุบัติเหตุ

ดึงข้อมูลสถิติอุบัติเหตุ (ปี 2568) จาก **MOT Data Catalog** (datagov.mot.go.th)
ผ่าน API แบบแบ่งหน้า (pagination) แล้วกรองเฉพาะพื้นที่ **กรุงเทพฯ และปริมณฑล 6 จังหวัด**
(กรุงเทพมหานคร นนทบุรี ปทุมธานี สมุทรปราการ นครปฐม สมุทรสาคร) พร้อมทั้ง
คัดกรองแถวที่พิกัดไม่สมบูรณ์ออก

### ขั้นที่ 2 — ประมวลผลและจัดกลุ่มจุดเสี่ยง

นำจุดอุบัติเหตุที่ผ่านการทำความสะอาดมาจัดกลุ่มด้วย **DBSCAN** (eps = 400 ม.,
min_samples = 3) ได้จุดเสี่ยงรวมประมาณ **108 จุด** จากนั้นสรุปคุณลักษณะเด่นของแต่ละกลุ่ม
ด้วยค่าฐานนิยม (mode) เช่น มูลเหตุที่พบบ่อย ลักษณะถนน รูปแบบการชน

### ขั้นที่ 3 — คำนวณคะแนนความเสี่ยงและแบ่งระดับ

คำนวณ `risk_score` (เต็ม 100) จาก 4 ปัจจัย (ความถี่ + ความรุนแรง EPDO + ลักษณะถนน
+ ความเร็ว) แล้วแบ่งเป็น 3 ระดับ (สูง/ปานกลาง/ต่ำ) ด้วยเกณฑ์ควอร์ไทล์ พร้อม KSI override
ผลลัพธ์บันทึกเป็นไฟล์ **GeoJSON** (`data/risk_points_bkk_metro.geojson`)
สคริปต์ทั้งหมดอยู่ใน `scripts/build_risk_points.py`

### ขั้นที่ 4 — พัฒนาเว็บแอปพลิเคชันแผนที่

สร้างเว็บหน้าเดียว (`index.html`) ด้วย **Leaflet + OpenStreetMap** โดยแบ่งโค้ดเป็นโมดูล:

| ไฟล์ | หน้าที่ |
|---|---|
| `js/distance.js` | คำนวณระยะทาง Haversine + bounding box |
| `js/riskpoints.js` | โหลด GeoJSON + วาดหมุดสีตามระดับความเสี่ยง |
| `js/map.js` | ตั้งค่าแผนที่ + หมุดตำแหน่งผู้ใช้ + auto-pan |
| `js/gps.js` | ติดตามตำแหน่ง (watchPosition) + โหมดจำลอง |
| `js/alert.js` | ตรวจรัศมี 500 ม. + cooldown กันเตือนซ้ำ |
| `js/tts.js` | เสียงพูดไทย + fallback แบนเนอร์ |
| `js/riskrules.js` | สร้างข้อความเตือนแบบ Dynamic Alert |
| `js/app.js` | จุดเริ่มต้น ประกอบทุกโมดูล |

### ขั้นที่ 5 — สร้าง REST API สำหรับอุปกรณ์ภายนอก

พัฒนาเซิร์ฟเวอร์ **FastAPI** (`api/server.py`) เปิด endpoint สำคัญ ได้แก่

| Endpoint | หน้าที่ |
|---|---|
| `GET /api/health` | สถานะเซิร์ฟเวอร์ + จำนวนจุดเสี่ยง |
| `GET /api/risk-points` | จุดเสี่ยงทั้งหมด (กรองตามระดับ/จังหวัด/คะแนนได้) |
| `GET /api/risk-points/nearby` | จุดในรัศมี เรียงใกล้→ไกล พร้อมข้อความเตือนสำเร็จรูป |
| `GET /api/risk-points/{id}` | รายละเอียดจุดเดียว |

### ขั้นที่ 6 — พัฒนาไคลเอนต์บน Raspberry Pi

เขียน `device/pi_alert_client.py` ที่วนทำงานทุก 3 วินาที: อ่านพิกัด GPS → เรียก API
`/nearby` → หากมีจุดเสี่ยงใกล้กว่า 500 ม. และยังไม่เคยเตือน → พูดข้อความเตือนผ่าน
`espeak-ng` โดยใช้ตรรกะ cooldown ชุดเดียวกับหน้าเว็บ รองรับ 3 โหมด:
GPS จริง (`--gpsd`), พิกัดคงที่ (`--test`), และจำลองเส้นทางจากไฟล์ (`--route`)

### ขั้นที่ 7 — ทดสอบระบบ

- **ทดสอบหน่วยย่อย (unit test):** ทดสอบฟังก์ชันคำนวณระยะทาง (`node tests/distance.test.js`, 9 เคส)
- **ทดสอบทั้งระบบ (โหมดจำลอง):** เปิด `http://localhost:8123/?mock=1` จำลองการขับรถ
  ~40 กม./ชม. วิ่งผ่านจุดเสี่ยงระดับสูง ต้องได้ยินเสียงเตือน **หนึ่งครั้ง** เมื่อเข้าเขต 500 ม.
  และไม่เตือนซ้ำจนกว่าจะออกนอกเขตแล้วกลับเข้ามาใหม่
- **ทดสอบอุปกรณ์:** รัน `pi_alert_client.py` ด้วยพิกัดคงที่/ไฟล์เส้นทาง เพื่อยืนยัน
  การเรียก API และการพูดเตือนก่อนติดตั้งบน GPS จริง
